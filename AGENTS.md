# Repository Guidelines

## Project Structure & Module Organization
The main application is a Laravel 12 project. Backend code lives in `app/`, with workflow engine code under `app/Engine`, reusable services in `app/Services`, HTTP entry points in `app/Http`, and workflow nodes in `app/Nodes`. Frontend assets live in `resources/js`, `resources/css`, and Blade views in `resources/views`. Routes are split across `routes/web.php`, `routes/api.php`, `routes/console.php`, and `routes/youtube.php`. Database migrations, factories, and seeders are in `database/`. Tests are organized by scope in `tests/Unit` and `tests/Feature`; private stabilization checks, when present, live under `tests/Feature/Stabilization` and can be run by path. The separate `mcp-server/` workspace contains TypeScript source in `mcp-server/src`; compiled output is generated by `npm run build` and is not part of the public export.

## Build, Test, and Development Commands
Use `composer install` and `npm install` for the main app, then `composer dev` to run the Laravel server, queue listener, log tail, and Vite together. Use `npm run dev` for frontend-only work and `npm run build` to create production assets. Run `composer test` for the default test pass, or `php artisan test --testsuite=Unit` / `php artisan test --testsuite=Feature` to target a suite. Private stabilization checks can be targeted by path with `php artisan test tests/Feature/Stabilization` when that directory is present. For the MCP server, use `cd mcp-server && npm install`, `npm run build`, `npm run watch`, and `npm start`.

## Coding Style & Naming Conventions
Follow `.editorconfig`: UTF-8, LF endings, spaces, and 4-space indentation by default; YAML uses 2 spaces. Keep PHP classes PSR-4 aligned with folder names under `App\\...`, and mirror existing naming such as `*Controller`, `*Service`, and `*Node`. Vue components use PascalCase filenames in `resources/js/src`, while composables use `useX.js`. Format PHP with `./vendor/bin/pint` before submitting. Do not commit generated build output unless it is explicitly tracked for release; public export omits generated `dist/` directories.

## Testing Guidelines
Add tests beside the relevant behavior and use descriptive `*Test.php` names, for example `WorkflowEngineTest.php` or `ScheduledJobCommandTest.php`. Prefer unit tests for isolated node and service logic, and feature or stabilization tests for full workflow, schema, and command paths. PHPUnit is configured through `phpunit.xml`; app source coverage is scoped to `app/`.
Schema-mutating tests must follow `docs/testing-isolation-playbook.md`: use `Tests\Support\PreservesSchemaTables`, a per-test sqlite connection, or a dedicated test database instead of permanently creating/dropping shared migrated tables.

## Commit & Pull Request Guidelines
Recent history uses Conventional Commit prefixes like `fix:`, `perf:`, and `docs:`. Keep commits focused and imperative, for example `fix: guard stale-file reindex loop`. For `fix:` commits that touch production behavior, include a body with root cause, behavior changed, verification, and any deployment or rollback notes; `.gitmessage` carries the local template. Pull requests should describe the user-visible or operational impact, list any config or migration changes, link the relevant issue, complete the production-behavior checklist when applicable, and include screenshots when UI views in `resources/views` or `resources/js` change.
